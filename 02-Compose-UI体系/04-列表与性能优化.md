# 列表与性能优化

> 新闻类 App 的核心就是「列表」。LazyColumn 是 Compose 的列表组件，性能优化是关键。

---

## 1. LazyColumn 基本用法

```kotlin
@Composable
fun NewsList(newsItems: List<NewsItem>, onItemClick: (NewsItem) -> Unit) {
    LazyColumn {
        items(
            items = newsItems,           // 数据列表
            key = { item -> item.id }    // 唯一 key（推荐！）
        ) { item ->
            NewsCard(news = item, onClick = { onItemClick(item) })
        }
    }
}
```

**为什么需要 key？** 如果数据顺序变了（如插入一条新闻），Compose 用 key 而不是 position 来判断哪个 item 变化了。没有 key 时，列表更新可能导致错误的重组或动画异常。

**后端类比**：key ≈ React 的 key prop ≈ 数据库主键，用于标识列表项的身份。

---

## 2. LazyColumn vs RecyclerView

| 对比 | RecyclerView | LazyColumn |
|------|------------|------------|
| 使用复杂度 | 需要 Adapter + ViewHolder + LayoutManager | 直接 `items {}` 即可 |
| 编译时检查 | XML id→findViewById→类型转换 | Kotlin 静态类型，编译期保证 |
| 复用机制 | 手动管理 ViewHolder 复用池 | 框架自动处理，key 保证 Diff 正确 |
| 动画 | 自己实现 ItemAnimator | 内置 `animateItem()` |
| 性能 | 成熟稳定，可手动微调 | 新框架，大部分场景够用 |

**面试问「为什么选 LazyColumn」时**：更简洁（不需 Adapter/ViewHolder）、类型安全（编译时检查）、与 Compose 的状态管理体系无缝集成。

## 3. key 的作用

```kotlin
// ❌ 没有 key — 顺序变化时可能错位
LazyColumn {
    items(newsItems) { item ->  // 靠索引匹配
        NewsCard(item)
    }
}

// ✅ 有 key — 通过 ID 匹配，顺序变化也无所谓
LazyColumn {
    items(newsItems, key = { it.id }) { item ->
        NewsCard(item)
    }
}
```

**没有 key 时的问题**：
1. 删除第一条数据后，第二条数据被当成第一条渲染
2. 状态（如展开状态）错位到其他 item
3. 不必要的重组（本该复用的 Composable 被重新创建）

---

## 4. 避免不必要的重组

```kotlin
// ❌ 问题：ListState 变化时，所有 NewsCard 都重组
val listState = rememberLazyListState()
LazyColumn(state = listState) {
    items(newsItems, key = { it.id }) { item ->
        NewsCard(item)  // 每个 item 都重组！
    }
}

// ✅ 优化：用 key 让 Compose 知道哪些 item 没变
// Compose 会自动跳过「数据没变」的 item
```

**进阶优化**：

```kotlin
// 把 NewsCard 标记为 @Stable，让 Compose 知道它只依赖不变参数
@Composable
fun NewsCard(news: NewsItem, onClick: () -> Unit) {
    // Compose 会比较 news 参数，不变就跳过重组
}
```

## 5. 图片加载性能

```kotlin
// 项目中用 Coil 加载图片
@Composable
fun NewsImage(imageUrl: String) {
    AsyncImage(
        model = ImageRequest.Builder(LocalContext.current)
            .data(imageUrl)
            .size(360, 240)                     // 限定尺寸，避免加载原图
            .precision(Precision.AUTOMATIC)     // 按需采样
            .build(),
        contentDescription = null,
        modifier = Modifier.fillMaxWidth()
    )
}
```

**与列表滑动的配合**：

```kotlin
// 方案：滑动时暂停图片加载（减少主线程压力）
val listState = rememberLazyListState()
val imageLoader = LocalContext.current.imageLoader

LaunchedEffect(listState.isScrollInProgress) {
    if (!listState.isScrollInProgress) {
        // 停止滑动时，恢复加载可见区域的图片
        imageLoader.resume()
    }
    // 滑动时 Coil 自动降低加载优先级
}
```

**Coil vs Glide 面试回答**：Coil 是 Kotlin 协程原生的图片加载库，更轻量，对 Compose 支持最好（有 `AsyncImage` composable）。Glide 是更成熟的方案，但基于 RxJava/线程池。新 Compose 项目推荐 Coil。

## 6. Modifier 的顺序

```kotlin
// ⚠️ Modifier 顺序很重要！修改的是链式处理顺序
Text("Hello",
    modifier = Modifier
        .padding(16.dp)       // 先加外边距
        .background(Color.Red) // 背景色只在 padding 之后的区域
)
// 结果：内层红色背景 + 外层 16dp 透明边距

Text("Hello",
    modifier = Modifier
        .background(Color.Red) // 先画背景
        .padding(16.dp)        // 后加边距（边距区域也是红色！）
)
// 结果：红色填充整个区域，内容和背景之间有 16dp
```

**口诀**：Modifier 是**从左到右**链式处理的，先写的在外层。

## 7. 性能优化清单

| 优化手段 | 效果 | 适用场景 |
|---------|------|------|
| 加 `key` | 避免错位重组 | 所有 LazyColumn |
| `size()` 限定图片尺寸 | 减少内存占用 | 列表中的图片 |
| `derivedStateOf` | 减少重组中的重复计算 | 有派生状态的场景 |
| `@Stable` / `@Immutable` | 编译器知道可以跳过重组 | 大量使用的 data class |
| `remember` 缓存 Modifier | 避免重复创建 | 任何 Composable |
| 滑动时降低图片加载优先级 | 保证滑动流畅 | 图片密集的列表 |

## 面试速记

| 问题 | 回答 |
|------|------|
| LazyColumn 为什么需要 key？ | 唯一标识每个 item，确保数据顺序变化时 Compose 能正确 Diff |
| LazyColumn 和 RecyclerView 的区别？ | LazyColumn 更简洁（无需 Adapter/ViewHolder），类型安全，与 Compose 集成好 |
| 列表卡顿怎么优化？ | 加 key、图片限定尺寸、滑动时降低加载优先级、@Stable 减少重组 |
| Modifier 顺序重要吗？ | 重要！从左到右链式处理，padding 在 background 前后的视觉效果不同 |
