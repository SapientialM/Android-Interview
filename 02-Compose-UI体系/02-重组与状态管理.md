# 重组与状态管理

> 面试重点 ⭐⭐⭐。「重组」是 Compose 面试的必考题，必须能讲清楚。
> **本质理解**：重组 ≈ React 的 re-render，数据变了就重新执行 Composable 函数。

---

## 1. 什么是重组 (Recomposition)？

**一句话**：当 Composable 函数依赖的 State 变化时，Compose 会**重新执行**这个函数来更新 UI。这个过程叫重组。

```kotlin
@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }  // State 对象
    Text("Count: $count")  // 读取 count → 建立了依赖关系
    Button(onClick = { count++ })  // 修改 count → 触发重组
}
```

**执行流程**：
1. 初始渲染：`count = 0` → `Text("Count: 0")`
2. 用户点击按钮：`count++` → count 从 0 变为 1
3. Compose 检测到 `count` 变化 → 重新执行 `Counter()` 函数
4. `Text("Count: 1")` — 屏幕更新

**关键机制：智能跳过**。Compose 编译期会在每个 Composable 调用前后插入参数比较（类似 `Object.equals`），如果所有参数都没变，就跳过这个 Composable 的重组。

**后端类比**：重组 ≈ 模板引擎重新渲染（Thymeleaf 在 model 变化时重新生成 HTML）。但 Compose 是增量更新（只更新变化的部分），而不是全量重建。

## 2. 状态管理四层模型

| 层级 | 工具 | 生命周期 | 适用场景 |
|------|------|---------|------|
| **composable 内** | `remember { mutableStateOf() }` | 重组之间 | 临时 UI 状态（如：输入框文字、展开/收起） |
| **composable 内** | `rememberSaveable { mutableStateOf() }` | 旋转屏幕、进程死亡 | 旋转屏幕后也要保留的状态 |
| **ViewModel** | `MutableStateFlow<T>` | ViewModel 生命周期 | 业务状态（如：新闻列表数据） |
| **全局** | `CompositionLocal` | 应用生命周期 | 主题、语言等全局配置 |

## 3. remember vs rememberSaveable

```kotlin
@Composable
fun SearchScreen() {
    // ❌ 每次重组都重置为空字符串
    var query = ""

    // ✅ 重组之间保留值
    var query = remember { mutableStateOf("") }

    // ✅ 旋转屏幕、进程死亡后也能恢复
    var query = rememberSaveable { mutableStateOf("") }
}

// 原理：remember 把值存在 Composable 的 Slot Table 里（内存中）
//       rememberSaveable 则额外通过 SavedStateHandle 序列化到 Bundle
```

**一句话选择**：
- 值不怕丢（如 Tab 选中索引）→ `remember`
- 值丢了用户有损失（如输入框内容、滚动位置）→ `rememberSaveable`

## 4. 状态提升 (State Hoisting)

**原则**：状态应该属于「需要它的最小组件的最近公共祖先」。

```kotlin
// ❌ 把状态下沉太低 — 父组件读不到子组件的状态
@Composable
fun SearchBar() {
    var query by remember { mutableStateOf("") }
    TextField(value = query, onValueChange = { query = it })
}
// 外面拿不到 query → 无法触发搜索

// ✅ 状态提升：状态在父组件，子组件接收 value + onValueChange
@Composable
fun SearchScreen() {
    var query by remember { mutableStateOf("") }

    SearchBar(
        query = query,                        // 向下传 value
        onQueryChange = { query = it }        // 向上传事件
    )

    SearchButton(onClick = {
        performSearch(query)                 // 父组件持有 query，可以触发搜索
    })
}

@Composable
fun SearchBar(query: String, onQueryChange: (String) -> Unit) {
    TextField(value = query, onValueChange = onQueryChange)
}
```

**状态提升的好处**：
- 单一数据源：状态只在一处定义
- 可复用：子组件变成「受控组件」
- 可测试：子组件只需要传参数，不需要 mock 状态

**后端类比**：状态提升 ≈ 把请求状态从 Service 层提升到 Controller 层管理，DAO 层只做无状态的 CRUD。

## 5. ViewModel + StateFlow — 生产中推荐的方式

```kotlin
// ViewModel — 持有 UI 状态
class NewsViewModel : ViewModel() {
    private val _uiState = MutableStateFlow(UiState())
    val uiState: StateFlow<UiState> = _uiState.asStateFlow()

    fun onQueryChange(query: String) {
        _uiState.update { it.copy(query = query) }
    }
}

// Composable — 只订阅状态，不做业务逻辑
@Composable
fun NewsScreen(viewModel: NewsViewModel = hiltViewModel()) {
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()

    SearchBar(
        query = uiState.query,
        onQueryChange = viewModel::onQueryChange
    )
}
```

**为什么 ViewModel 中的状态不丢？**
- Activity 旋转屏幕时被销毁重建
- 但 ViewModel 存在 ViewModelStore 中，重建的 Activity 可以取回同一个 ViewModel
- 这就像 `@SessionScope` Bean 在 HTTP Session 存活期间一直保留

## 6. Compose 的三个生命周期阶段

1. **Composition（组合）**：Composable 第一次执行，创建 UI 树
2. **Recomposition（重组）**：State 变化，重新执行 Composable 刷新 UI
3. **Disposal（销毁）**：Composable 离开 UI 树，释放资源

```kotlin
@Composable
fun LifecycleDemo() {
    val scope = rememberCoroutineScope()

    // Composition 时启动协程，key 变化或 Disposal 时取消
    LaunchedEffect(Unit) {
        // 这是副作用（网络请求等）
    }

    // Disposal 时清理资源
    DisposableEffect(Unit) {
        val listener = Listener()
        register(listener)
        onDispose {
            unregister(listener)  // 清理
        }
    }
}
```

## 7. 重组的陷阱与优化

### 陷阱一：在高频变化的 Composable 中创造对象

```kotlin
// ❌ 每次重组都创建新的 Modifier 对象
@Composable
fun BadItem(item: Item) {
    Text(
        text = item.title,
        modifier = Modifier.padding(16.dp)  // 每次重组都新创建
    )
}

// ✅ 用 remember 缓存
@Composable
fun GoodItem(item: Item) {
    val modifier = remember { Modifier.padding(16.dp) }
    Text(
        text = item.title,
        modifier = modifier
    )
}
```

### 陷阱二：不稳定参数导致无效重组

```kotlin
// ❌ data class 没有被标记为 @Stable，Compose 默认认为它总是变化的
data class User(val name: String, val age: Int)

// ✅ 加 @Stable 或 @Immutable 告诉 Compose 它只在字段变化时才变
@Stable
data class User(val name: String, val age: Int)
```

**`@Stable` vs `@Immutable`**：
- `@Stable`：字段可能变，但 Compose 能追踪到变化（MutableState）
- `@Immutable`：字段保证不可变（val），一旦创建就永远不会变

## 面试速记

| 考点 | 回答要点 |
|------|---------|
| 重组 | State 变化 → Composable 重新执行 → 智能跳过（参数比较） |
| 状态管理层次 | remember（重组间保持）→ rememberSaveable（配置变更保持）→ ViewModel+StateFlow（业务状态） |
| 状态提升 | 状态在父组件，子组件通过参数接收（类比 React lift state up） |
| 重组优化 | @Stable/@Immutable、derivedStateOf、LazyColumn 加 key |
| 为什么网络请求不能放 Composable | 因为每次重组都会重新执行，导致重复请求 |
