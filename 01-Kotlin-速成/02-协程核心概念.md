# 协程核心概念

> 面试重点 ⭐⭐⭐。协程是 Kotlin 面试的必考点，也是你项目中网络请求的底层依赖。
> **核心策略**：用 Java 并发 API 类比学习。

---

## 1. 协程到底是什么？

**一句话**：协程是**用户态的轻量级线程**，可以在不阻塞线程的情况下挂起和恢复。

```kotlin
// 这段代码在主线程执行
fun loadData() {
    viewModelScope.launch {
        // 下面这行会「挂起」协程，但不会阻塞主线程
        val result = withContext(Dispatchers.IO) {
            api.getData()  // 网络请求在 IO 线程
        }
        // 网络请求完成后，自动回到主线程继续执行
        updateUI(result)
    }
}
```

**后端类比**：

| 概念 | Java 后端 | Kotlin 协程 |
|------|----------|-----------|
| 异步任务 | `CompletableFuture.supplyAsync()` | `launch { }` / `async { }` |
| 等待结果 | `future.get()` (阻塞) | `await()` (挂起，不阻塞线程) |
| 线程切换 | `thenApplyAsync(fn, executor)` | `withContext(Dispatchers.IO) {}` |
| 生命周期绑定 | Bean 销毁时的线程清理 | `viewModelScope` / `lifecycleScope` |
| 结构化并发 | StructuredTaskScope (Java 21) | `coroutineScope {}` |

## 2. suspend 函数的本质

```kotlin
suspend fun fetchData(): String {
    delay(1000)  // 挂起 1 秒，不阻塞线程
    return "data"
}
```

**面试口述**：
- `suspend` 是编译器关键字，标记的函数可以被「挂起」
- 本质上编译后会多一个 `Continuation` 参数（CPS 转换，Continuation Passing Style）
- 挂起 ≠ 阻塞：挂起是保存当前状态然后让出线程，阻塞是占着线程等待
- 协程可以在一个线程上运行多个，挂起时线程去做别的事

**后端类比**：`suspend` 函数 ≈ Java 21 虚拟线程的阻塞 IO — 虽然代码看起来是同步的，但实际不阻塞底层线程。

## 3. launch vs async

```kotlin
// launch — 不返回结果，返回 Job
val job = viewModelScope.launch {
    // 执行一些操作
    api.sendLog(event)  // 不需要返回值
}

// async — 返回结果，返回 Deferred（类似 Future）
val deferred = viewModelScope.async {
    api.getNews()  // 需要返回值
}
val news = deferred.await()  // 挂起等待结果
```

| 对比维度 | launch | async |
|---------|--------|-------|
| 返回值 | Job（无结果） | Deferred\<T\>（有结果） |
| 后端类比 | `runAsync()` | `supplyAsync().get()` |
| 异常处理 | 直接抛出 | await() 时才抛出 |
| 适用场景 | fire-and-forget | 需要一个返回值 |

## 4. 调度器 (Dispatchers)

```kotlin
viewModelScope.launch(Dispatchers.Main) {
    // 主线程 — 更新 UI
    val data = withContext(Dispatchers.IO) {
        // IO 线程池 — 网络请求、文件读写
        api.fetchData()
    }
    val processed = withContext(Dispatchers.Default) {
        // CPU 线程池 — 密集计算
        processData(data)
    }
    updateUI(processed)  // 回到 Dispatchers.Main
}
```

| 调度器 | 后端类比 | 使用场景 |
|--------|---------|------|
| `Dispatchers.Main` | 主线程/EventLoop 线程 | UI 更新 |
| `Dispatchers.IO` | 弹性线程池（类似 CachedThreadPool） | 网络、文件、数据库 |
| `Dispatchers.Default` | 固定线程池（CPU 核数个线程） | 计算密集型（JSON 解析、排序） |
| `Dispatchers.Unconfined` | — | 极少用，不限定线程 |

**面试常问**：`withContext(Dispatchers.IO)` 和 `flowOn(Dispatchers.IO)` 的区别？
- `withContext`：切换协程的执行线程，块代码在指定线程执行
- `flowOn`：切换 Flow 的 emit 线程，只影响上游

## 5. 结构化并发

```kotlin
// coroutineScope — 创建一个结构化并发作用域
suspend fun loadTwoDataSources(): Pair<String, String> = coroutineScope {
    val deferred1 = async { api.getData1() }
    val deferred2 = async { api.getData2() }
    // 如果任何一个 async 失败，scope 内所有协程会被取消
    Pair(deferred1.await(), deferred2.await())
}
// coroutineScope 完成后，保证所有子协程都已完成
```

**结构化并发的三个保证**：
1. 父协程会等待所有子协程完成
2. 父协程取消时，子协程自动取消
3. 子协程异常会向上传播（除非用 SupervisorJob）

**后端类比**：结构化并发 ≈ Java 21 的 `StructuredTaskScope`（JEP 453）— 确保生成的任务在退出作用域前完成。

## 6. SupervisorJob vs Job

```kotlin
// 默认 Job — 一个子协程失败，所有兄弟协程被取消
viewModelScope.launch {
    launch { throw Exception() }  // 这个失败...
    launch { api.getData() }      // ...会导致这个也被取消
}

// SupervisorJob — 子协程失败不影响兄弟协程（典型用途：viewModelScope）
// viewModelScope 默认使用 SupervisorJob
viewModelScope.launch {
    launch { api.fetchNews() }   // 即使这个失败...
    launch { api.fetchBanner() } // ...这个仍然执行
}
```

**为什么 ViewModel 用 SupervisorJob？** 因为新闻列表加载失败不应导致 Banner 也加载失败。

## 7. 协程异常处理

```kotlin
// 方式一：try-catch（适用于 async/await）
viewModelScope.launch {
    try {
        val data = withContext(Dispatchers.IO) {
            api.getData()
        }
        updateUI(data)
    } catch (e: Exception) {
        showError(e.message)
    }
}

// 方式二：CoroutineExceptionHandler（适用于 launch）
val handler = CoroutineExceptionHandler { _, exception ->
    // 全局捕获未被处理的异常
    logError(exception)
}
viewModelScope.launch(handler) {
    api.getData()  // 如果这里抛异常，handler 会收到
}
```

**重要规则**：
- `launch` 的异常会立即抛出，用 `CoroutineExceptionHandler` 或 `try-catch` 处理
- `async` 的异常在 `await()` 时才抛出，包在 `try-catch` 里就行
- `SupervisorJob` 下的子协程失败不会取消其他子协程

## 8. 项目中的典型用法

```kotlin
class NewsViewModel @Inject constructor(
    private val repository: NewsRepository
) : ViewModel() {

    private val _uiState = MutableStateFlow<UiState>(UiState.Loading)
    val uiState: StateFlow<UiState> = _uiState.asStateFlow()

    fun loadNews() {
        // viewModelScope：与 ViewModel 生命周期绑定
        // ViewModel 清理时自动取消协程
        viewModelScope.launch {
            _uiState.value = UiState.Loading
            try {
                val news = withContext(Dispatchers.IO) {
                    repository.getNews()
                }
                if (news.isEmpty()) {
                    _uiState.value = UiState.Empty
                } else {
                    _uiState.value = UiState.Success(news)
                }
            } catch (e: Exception) {
                _uiState.value = UiState.Error(e.message ?: "未知错误")
            }
        }
    }
}
```

## 面试速记

| 考点 | 回答要点 |
|------|---------|
| 协程 vs 线程 | 协程是用户态轻量级，挂起不阻塞线程，一个线程可以运行大量协程 |
| suspend | 编译器 CPS 转换，挂起=保存状态+让出线程，恢复=读取状态+继续执行 |
| launch vs async | launch 返回 Job（无结果），async 返回 Deferred（有结果，await 获取） |
| 调度器 | Main(UI)、IO(网络/文件)、Default(CPU计算) |
| 结构化并发 | 父等子、父取消子跟随、异常向上传播 |
| SupervisorJob | 子协程异常不影响兄弟（viewModelScope 的本质） |
