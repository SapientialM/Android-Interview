# 项目介绍模板

> 面试中用「痛点-方案-亮点」结构讲项目，不要流水账。

---

## 模板一：1 分钟版（快速介绍）

```
适用场景：HR 初筛、电话面试、被问「简单做个自我介绍」时顺带项目。

结构：
  1 句话背景 + 技术栈 + 我的角色 + 核心贡献
```

**示例**：

> 我独立开发了一个类今日头条的新闻资讯客户端，使用 Kotlin + Jetpack Compose + MVVM 架构。项目采用单向数据流设计：UI 层通过 StateFlow 订阅状态，ViewModel 处理业务逻辑，Retrofit + OkHttp 负责网络请求。我重点做了列表滑动性能优化和统一的状态管理方案。

---

## 模板二：3 分钟版（标准介绍）

```
适用场景：技术面试、答辩环节。

结构：
  背景 (20s) → 技术栈 (30s) → 架构设计 (60s) → 技术亮点/难点 (60s) → 总结 (10s)
```

**示例**：

> **背景**：这个项目是我参加的字节跳动工程训练营的课题，目标是用 Compose 技术栈实现一个类今日头条的资讯客户端。我独立负责了整个 Android 端的开发。
>
> **技术栈**：我选用了 Kotlin + Jetpack Compose + MVVM + Hilt 依赖注入 + Retrofit 网络请求 + Coil 图片加载。之所以选 Compose 而不是传统 View，是因为声明式 UI 更契合 Kotlin 的协程和响应式特性，而且也是 Google 的趋势方向。
>
> **架构设计**：我采用了单向数据流设计，整体分为三层：UI 层（Compose Screen）只负责渲染和事件传递；ViewModel 层持有 StateFlow 状态，处理业务逻辑；Repository 层聚合远程和本地数据源。这样做的好处是状态管理清晰、便于后续单元测试。
>
> **技术难点**：过程中我重点解决了两个难点。一是首页列表是无限滚动的信息流，我通过 LazyColumn 的 key 复用、Coil 按需加载和缓存策略保证了滑动流畅性；二是 Compose 的重组机制，如果状态管理不好会造成无效重组，我通过把状态统一上提到 ViewModel，用 sealed class 定义 UiState，避免了在 Composable 中直接做业务逻辑。
>
> **总结**：通过这个项目，我从零学会了 Kotlin 和 Compose，理解了移动端的架构设计思想，也体会到了声明式 UI 带来的开发效率提升。

## 模板三：5 分钟版（详细答辩）

```
适用场景：正式答辩、终面。

结构：
  背景 (30s) → 技术栈+选型理由 (60s) → 架构设计+数据流 (90s) → 
  技术亮点/难点×2 (120s) → 学到什么+展望 (30s)
```

在 3 分钟版基础上增加：
1. **技术选型对比**：为什么选 Compose 不是传统 View？为什么 Coil 不是 Glide？
2. **架构图展示**：PPT 里的架构图翻出来配合讲
3. **具体代码案例**：挑一段核心代码讲（如 ViewModel 的状态管理、Repository 的异常处理）
4. **如果有 Demo**：0.5 分钟快速滑动演示
5. **更多 Q&A 预判**：在「总结」后说「欢迎大家提问」

## 常见雷区（千万不要）

| 雷区 | 为什么不好 | 正确做法 |
|------|-----------|---------|
| 「我只是负责写页面」 | 显得没有技术含量 | 讲架构、讲难点、讲为什么这样设计 |
| 「大多数代码是 AI 写的」 | 面试官会质疑你的技术能力 | 说「AI 帮我生成模板代码，但架构设计、技术选型、性能优化和边界处理是我自己做的」 |
| 只讲功能不讲技术 | 「登录、列表、详情、分享」这谁不会？ | 每个功能对应一个技术决策 |
| 夸大团队贡献 | 容易被追问穿帮 | 诚实，但强调你负责的核心部分 |
| 花太多时间在功能演示 | 面试官在意的是技术思考 | 功能一句话带过，技术讲详细 |
