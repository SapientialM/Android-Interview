# 技术难点答辩模板

> 面试中关于「技术难点」的问题是最容易拉开分差的环节。
> 核心策略：准备 3 个「技术故事」，每个都能完整讲 2 分钟。

---

## 常见问法及回答套路

### 问法一：「你遇到的最大技术挑战是什么？」

**套路：背景 → 问题 → 尝试 → 方案 → 结果（STAR 法则）**

```
背景 (Situation)：项目是什么，当时在做什么功能
问题 (Problem) ：具体遇到了什么问题
尝试 (Attempt) ：一开始怎么尝试的，为什么不够好
方案 (Action)  ：最终采用了什么解决方案，为什么选它
结果 (Result)  ：优化了多少（量化），学到了什么
```

**示例（列表卡顿优化）**：

> **背景**：我在开发新闻客户端首页时，信息流列表需要展示大量带图片的新闻卡片。
>
> **问题**：快速滑动时出现明显的卡顿和图片加载闪烁。我通过 Layout Inspector 发现有大约 40% 的帧超过了 16ms 的渲染预算。
>
> **尝试**：一开始我以为是图片太多，试着把所有图片都预加载到内存缓存。但效果不好，因为预加载大量图片反而导致 GC 频繁触发。
>
> **方案**：我做了三个层面的优化：一是给 LazyColumn 的每个 item 指定了稳定的 key，让 Compose 能正确做 Diff；二是用 Coil 的 size() 限定只解码到 ImageView 的实际尺寸，而不是加载原图；三是在快速滑动时通过 LaunchedEffect 暂停图片加载，等用户停止滑动再恢复。
>
> **结果**：优化后滑动流畅度提升明显，丢帧率从 40% 降到了 5% 以下。

### 问法二：「为什么选择这个技术方案而不是另一个？」

**套路：方案 A 优缺点 → 方案 B 优缺点 → 为什么选 A**

**示例（Compose vs XML）**：

> 我在技术选型时比较了 Compose 和传统 XML View 方案。传统 XML 方案成熟、资料多、生态稳定，但需要手动写 Adapter、ViewHolder、findViewById，大量样板代码。Compose 是声明式 UI，用 Kotlin 写 UI，状态驱动更新，代码量更少，Google 也在全力投入这个方向。考虑到这是一个新项目而且 Compose 的 API 已经稳定，我选了 Compose。如果是一个大型存量项目，可能会考虑逐步迁移而不是全新用 Compose。

---

### 问法三：「如果让你重新设计，你会怎么改进？」

**套路：当前方案不足 → 理想方案 → 成本考量**

**示例**：

> 如果重新做，我会考虑三个方向。一是在架构层面，当前项目比较小没有引入 MVI 模式，如果交互更复杂（如搜索+多级筛选），我会把 Intent 封装成 sealed class 统一管理。二是在数据层，目前只有网络请求没有本地缓存，离线体验不好，我会引入 Room 做离线缓存和 DataStore 存储用户偏好。三是在测试方面，目前没有写单元测试，我会给 ViewModel 和 Repository 补上单测。

## 准备 3 个技术故事

选你项目中最有亮点的 3 个技术点，每个用 STAR 法则写一个故事：

| 故事主题 | 你的具体经历 |
|---------|------------|
| 架构设计 | 为什么选 MVVM + 单向数据流？怎么分层的？ |
| 性能优化 | 列表/图片/启动/包体积中具体做了哪个优化？ |
| 边界处理 | 弱网/空状态/异常/旋转屏幕中怎么处理的？ |

---

## 面试时的注意事项

1. **不要只说「我没遇到什么困难」**：这会显得你没做任何复杂的事
2. **不要编造故事**：会被追问穿帮。说真实的，如果不炫就说「现有的难点是 X，我计划用 Y 方案优化」
3. **量化结果**：不要说「变快了」，要说「丢帧率从 40% 降到 5%」
4. **如果被追问细节**：不要慌，诚实说你知道的边界。不知道的部分说「这块我也在学」
