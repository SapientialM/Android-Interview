# 模块化设计

> 面试进阶题 ⭐。小项目不需要模块化，但面试中提一句能体现工程思维。
> **后端类比**：Android 模块化 ≈ Maven 多模块项目。

---

## 1. 为什么要模块化？

| 好处 | 说明 |
|------|------|
| 编译速度 | 只改一个模块就不用全量编译 |
| 团队协作 | 不同团队负责不同模块，减少 Git 冲突 |
| 代码隔离 | 通过 `implementation`/`api` 控制依赖可见性 |
| 可替代 | 换一个功能实现只影响一个模块 |

---

## 2. 常见模块划分

```
app/                        ← 主模块（入口、DI、导航）
│
├─ :core:network            ← 网络层（OkHttp + Retrofit）
├─ :core:data               ← 数据层（Repository 接口 + DTO 定义）
├─ :core:ui                 ← 公共 UI 组件（LoadingView, ErrorView）
├─ :core:model              ← 公共数据模型（Domain Model）
├─ :core:common             ← 工具函数、扩展函数
│
├─ :feature:news-list       ← 新闻列表功能
├─ :feature:news-detail     ← 新闻详情功能
├─ :feature:settings        ← 设置页
│
└─ :feature:...             ← 更多功能模块
```

**后端类比**：
- `app` ≈ 主 Spring Boot 工程
- `:core:network` ≈ `common-http-client` 共享网络库
- `:feature:xxx` ≈ 微服务或按领域拆分的模块

## 3. api vs implementation

```kotlin
// 模块 A 的 build.gradle.kts
dependencies {
    implementation(project(":core:network"))
    // implementation：core:network 的 API 只在 A 内部可见
    // 模块 B 依赖 A 时，看不到 core:network 的类

    api(project(":core:model"))
    // api：core:model 的 API 对依赖 A 的模块也可见
    // 模块 B 依赖 A 时，可以直接使用 core:model 的类
}
```

**选择原则**：默认用 `implementation`，只有当模块的公开接口需要暴露上游依赖时才用 `api`。

## 4. 模块间通信

```kotlin
// 方式一：接口下沉（推荐）
// :core:data 中定义 Repository 接口
interface NewsRepository {
    suspend fun getNews(): List<NewsItem>
}
// :feature:news-list 依赖 :core:data，实现自己的 Repository
// :app 通过 DI 注入具体实现

// 方式二：依赖注入（Hilt）
// 每个 feature 模块有自己的 Module，app 模块通过 interface + @Binds 组装

// 方式三：Navigation 跳转
// Deep Link 实现跨模块跳转
navController.navigate("feature://detail/$newsId")
```

## 5. 面试话术

> 「我在项目中没有做模块化，因为功能还不够多。但如果后续扩展，我会考虑拆分为：
> `app` 主模块、`:core:network` 网络层、`:core:ui` 公共组件、
> `:feature:news-list` 新闻列表、`:feature:news-detail` 新闻详情。
> 通过 `implementation` 控制依赖可见性，
> 接口下沉到 core 层实现模块解耦，
> Hilt 的 `@Binds` 将各模块的实现注入到主模块。」

## 面试速记

| 问题 | 回答 |
|------|------|
| 你的项目怎么分模块的？ | 项目不大所以没拆分，但设计了 feature/core 分层（展示你的分层意识） |
| api vs implementation | api 暴露给下游，implementation 仅内部可见（和 Gradle 的 api/implementation 概念一致） |
