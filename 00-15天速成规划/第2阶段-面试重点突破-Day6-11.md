### 下午：Activity 深度学习（3h）

阅读 [05-面试八股/03-Activity生命周期与启动模式](../05-面试八股/03-Activity生命周期与启动模式.md)

**必背要点**：

| 考点 | 面试口述要点 |
|------|-------------|
| 生命周期 | onCreate → onStart → onResume → onPause → onStop → onDestroy |
| A 启动 B 回调顺序 | A.onPause → B.onCreate → B.onStart → B.onResume → A.onStop |
| 对话框弹出 | 不会触发 onStop（Activity 仍可见） |
| onSaveInstanceState | 系统回收 Activity 前调用，用于保存临时状态 |
| ViewModel 存活原因 | 关联 Activity 的 ViewModelStore，只有 finish 才清理 |

**四种启动模式**：

| 模式 | 行为 | 应用场景 |
|------|------|---------|
| Standard | 每次新建实例 | 默认 |
| SingleTop | 栈顶复用（onNewIntent） | 通知点击进入已打开的页面 |
| SingleTask | 栈内复用，清掉上面的 | 主页/首页 |
| SingleInstance | 独占一个栈 | 极少使用，如电话拨号 |

**后端类比**：Activity 生命周期 ≈ Servlet/Filter 生命周期回调。ViewModel 存活 ≈ @SessionScope Bean 在 Session 有效期内不销毁。

### 下午：Handler 源码级理解（3h）

阅读 [05-面试八股/01-Handler消息机制](../05-面试八股/01-Handler消息机制.md)

**必背要点**：

```
Handler 四件套：
  Looper         — 消息循环（类比 EventLoop）
    └─ MessageQueue  — 消息队列（类比 BlockingQueue）
       ├─ Message      — 消息体
       └─ Handler      — 发送/处理消息

流程：
  Looper.prepare() → 创建 MessageQueue 并绑定到 ThreadLocal
  Looper.loop()    → 死循环取消息（queue.next()）
  Handler.sendMessage() → 消息入队（msg.target = handler）
  Looper 取出消息 → msg.target.dispatchMessage(msg) → handleMessage()
```

**面试常问**：「为什么主线程 Looper 死循环不会卡死？」

答：`nativePollOnce` 使用 Linux epoll 机制，没消息时线程阻塞/休眠，不消耗 CPU。有消息时唤醒。这和后端 Netty 的 EventLoop 用 epoll 监听 IO 事件完全一致。

阅读 [05-面试八股/02-Binder跨进程通信](../05-面试八股/02-Binder跨进程通信.md)

**面试版回答框架**（3 层递进）：

```
第 1 层（应用层）：AIDL 定义接口 → Proxy/Stub 自动生成 → 调用就像本地方法
第 2 层（Framework）：ServiceManager 是注册中心 → Client 通过 SM 找到 Server → Binder 驱动做中转
第 3 层（内核层）：Binder 用 mmap 实现一次拷贝（传统 IPC 需要两次：用户态→内核态→用户态）
```

**后端类比**：Binder ≈ gRPC/Thrift，ServiceManager ≈ Nacos/Eureka 注册中心，mmap ≈ 零拷贝/sendfile。

### 下午：协程/Flow（3h）

阅读 [05-面试八股/05-协程与Flow](../05-面试八股/05-协程与Flow.md)

**必背要点**：

| 概念 | 面试口述 |
|------|---------|
| 协程本质 | 用户态轻量级线程，suspend = 暂停 + 恢复，不阻塞线程 |
| launch vs async | launch 返回 Job（无结果），async 返回 Deferred（有结果，await 获取） |
| Dispatchers | Main（UI）、IO（网络/文件）、Default（CPU）、Unconfined |
| SupervisorJob | 子协程异常不影响兄弟协程 |
| StateFlow vs SharedFlow | StateFlow 热流（保留最新值，类似 BehaviorSubject）、SharedFlow 热流（无保留值，类似 PublishSubject） |
| Flow 冷流 | 每次 collect 重新执行，不 collect 就不执行 |

**后端类比**：协程 ≈ Java 21 虚拟线程，Flow ≈ Project Reactor Flux，StateFlow ≈ BehaviorSubject。
