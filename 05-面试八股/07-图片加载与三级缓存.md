# 图片加载与三级缓存

> 级别：⭐⭐ 高频。Android 面试常见题。

---

## 面试官怎么问

- 「图片三级缓存是哪三级？」
- 「Glide/Coil 的加载流程？」
- 「一张 1000×1000 的 ARGB_8888 图片占多少内存？」
- 「大图加载怎么处理？」

---

## 回答框架

### 1. 三级缓存

```
┌─────────────┐
│  内存缓存     │ ← LruCache（最近最少使用），强引用 + 弱引用
│  最快       │   Glide 默认使用 LruResourceCache
├─────────────┤
│  磁盘缓存     │ ← DiskLruCache，文件系统
│  较快       │   Glide 用 DiskLruCache
├─────────────┤
│  网络/源文件  │ ← 原始 URL / 本地文件
│  最慢       │
└─────────────┘
```

**后端类比**：内存缓存 ≈ Redis / Caffeine，磁盘缓存 ≈ MySQL / 文件缓存，网络 ≈ 外部 API。

### 2. Glide 加载流程（Coil 类似）

```
Glide.with(context)
    .load(url)
    .into(imageView)

流程：
1. with() → 绑定生命周期（Activity/Fragment 销毁时自动取消）
2. load() → 解析数据源类型（URL/File/Resource）
3. DecodeJob 执行：
   a. 检查内存缓存 → 命中
   b. 检查磁盘缓存 → 命中
   c. 网络下载 → 解码 → 存储到磁盘缓存 → 存储到内存缓存
4. into() → 绑定到 ImageView（或 Compose 的 AsyncImage）
```

### 3. Bitmap 内存计算

```
Bitmap 内存 = 宽 × 高 × 像素字节数（format）

ALPHA_8    = 1 字节/像素（只有透明度）
RGB_565    = 2 字节/像素
ARGB_8888  = 4 字节/像素（默认，最常用）
RGBA_F16   = 8 字节/像素

示例：一张 1000×1000 的 ARGB_8888 图片
  内存 = 1000 × 1000 × 4 = 4,000,000 字节 ≈ 3.8 MB
```

**面试常问**：「如果你在列表中展示了 10 张这样的图，占多少内存？」40 MB，这已经快接近单个 App 的内存限制了（通常 128-256 MB）。

### 4. 图片优化策略

| 策略 | 说明 |
|------|------|
| **按需采样** | `inSampleSize = 2` → 长宽各缩小 2 倍，内存降为 1/4 |
| **限定尺寸** | Coil/Glide 的 `size(w, h)` 只解码到所需尺寸 |
| **滑动暂停** | 快速滑动时暂停加载，停止后再加载可见区域 |
| **格式优化** | 不透明图片用 RGB_565（2 字节/像素）代替 ARGB_8888（4 字节） |
| **ARTHook 检测大图** | 运行时检测 Bitmap 是否大于 ImageView 尺寸 |

### 5. Coil（Compose 项目推荐）

Coil 是 Kotlin 协程原生的图片加载库：

```kotlin
AsyncImage(
    model = ImageRequest.Builder(LocalContext.current)
        .data(imageUrl)
        .size(360, 240)                    // 只加载所需尺寸
        .precision(Precision.AUTOMATIC)    // 自动采样
        .crossfade(true)
        .build(),
    contentDescription = null
)
```

**为什么 Compose 项目选 Coil 而不是 Glide？** Coil 是 Kotlin 协程原生的，更轻量，Compose 集成最好。Glide 基于线程池，功能多但较重。

## 后端类比

| 图片缓存 | 后端 |
|---------|------|
| 内存缓存 (LruCache) | Redis / Caffeine |
| 磁盘缓存 (DiskLruCache) | 数据库查询结果缓存 |
| 网络 | 外部 API 调用 |
| 采样 (inSampleSize) | 分页查询（限制返回条数） |
| Coil/Glide | Spring Cache Abstraction + WebClient |

## 延伸追问

1. **LruCache 的原理？**
   → 基于 LRU 的 HashMap。access 时把 entry 移到队尾，空间不足时从队头淘汰。Glide 使用 LinkedHashMap 实现。

2. **Glide 和 Picasso 的区别？**
   → Glide：支持 GIF、更多解码格式、内存占用更小（RGB_565 默认）、生命周期感知。Picasso：更轻量、API 更简。现在基本都用 Glide 或 Coil。

3. **Coil 和 Glide 怎么选？**
   → Compose 项目用 Coil（原生 Compose 支持）。传统 View 项目继续用 Glide。

4. **大图加载（如 5000×5000）怎么处理？**
   → `BitmapRegionDecoder` 分块解码，只加载当前可见区域。类似地图加载：Google Maps 把地图切成 tile。
